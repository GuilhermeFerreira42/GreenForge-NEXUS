Vulnerabilidade 1 – SSE + WebSocket Reorder Buffer

- **FINDING:** A biblioteca Fastify SSE oferece suporte a replay de eventos usando o cabeçalho `Last-Event-ID`. No exemplo de código, o manipulador SSE invoca `reply.sse.replay(async lastEventId => { … })` para reenviar mensagens perdidas após um reconexão baseada no último ID recebido【129†L468-L478】. Ou seja, ele mantém um histórico de mensagens (com IDs) e, ao receber o ID do último evento do cliente, procura mensagens subsequentes nesse histórico e as reenfilera para o cliente【129†L468-L478】. Isso resolve o gap detection aguardando explicitamente eventos faltantes e retransmitindo-os após reconexão.
- **FINDING:** O cliente OpenHands WebSocket usa um parâmetro de query `latest\_event\_id` na conexão para indicar o ID do último evento recebido. A documentação oficial explica que o servidor deve enviar todos os eventos subsequentes a esse ID na nova conexão【109†L1-L4】. Assim, o cliente pode reconectar informando o último `seq` conhecido (ou -1 para novo) e o servidor retorna apenas eventos a partir daquele ponto, tratando gaps em ordem.
- **FINDING:** Não foi encontrada implementação pública para mesclar streams SSE e WebSocket em ordem única. Nenhuma biblioteca ou projeto pesquisado oferece um buffer genérico para integrar canais SSE e WS simultâneos. Em suma, não há “reorder buffer” pronto de produção; seria necessário implementar logicamente uma fila em cada cliente que aguarde eventos perdidos por timeout configurável antes de avançar.

Vulnerabilidade 3 – Garbage Collection x Janela de Rollback

- **FINDING:** Não foram encontrados padrões prontos que implemente coleta de lixo (GC) de worktrees com sincronização com janelas de rollback. Em projetos de CI/CD consultados (Gitpod, Coder etc.), não há exemplos de checar referências antes de deletar. A ausência de “lock” entre GC e rollback é uma lacuna. *EVIDENCE:* buscas por termos como `canDelete`, `isEligibleForGC` ou `lease expiry rollback` não retornaram código relevante – não há exemplo de recurso sendo protegido via refcount ou flag durante janela de rollback.
- **FINDING:** Em repositórios auditados (Plandex, OpenHands), não existem funções específicas para arquivar planos ou branches com checagem de operações ativas. Não há “isActive” ou lógica de “lock” no código fonte público destas ferramentas para prevenir GC durante rollback. Portanto, conclui-se que nenhuma dessas implementações cuida deste caso – o padrão seguro exigiria adicionar uma contagem de referências ou sinalização de uso no gerenciador de rollback, algo inexistente.
- **FINDING:** Não foi identificado código em TypeScript que demonstre gerenciamento de recursos (diretório, container, branch) protegido por contagem de referências ou “leases”. Pesquisas em GitHub usando `refcount cleanup worktree TypeScript` ou `canDelete mergedAt` não retornaram projetos relevantes. Em resumo, ausência de padrão conhecido: normalmente teria-se um método `canDelete()` que consulta se há locks ativos, mas nenhum repositório estudado mostrou isso.

Vulnerabilidade 5 – Fingerprinting Semântico de Loops

- **FINDING:** Não foram encontrados implementações prontas em repositórios de AI para normalizar AST antes de hashing. Procura por termos como `ast normalize hash TypeScript` ou `simhash code deduplication` não revelou código de uso em ferramentas estudadas. A maioria das soluções indicadas (AST-diff, difftastic, tree-sitter) trata de diff visual, não hashing fixo. Assim, não existe padrão público de “hash semântico” incorporado a essas ferramentas; qualquer sistema terá de analisar o AST, renomear identificadores e hash do novo AST manualmente.
- **FINDING:** Em ferramentas conhecidas (Sourcegraph, Pyright, tree-sitter, evals), não há um utilitário direto de “hash semântico” de código. Tree-sitter pode gerar ASTs mas não fornece hash canônico; projetos como SemanticDiff ou difftastic são focados em diff interativo. A busca por `structural hash AST rename invariant` não retornou bibliotecas TypeScript maduras para produzir hashes invariantes de formatação. Nenhuma solução “battle-tested” específica foi encontrada, confirmando que o detector atual em GreenForge precisaria de desenvolvimento próprio ou uso de biblioteca genérica de AST (como esmorph ou babel) para normalização antes de hash.
- **FINDING:** Na pipeline de edições do Aider (paul-gauthier/aider), não existe código explícito para detectar e impedir sugestões duplicadas baseadas em hash. A pesquisa em `paul-gauthier/aider` não revelou nenhum `LoopDetector`, cache de sugestões ou lógica de deduplicação no código fonte, sugerindo que essa ferramenta não normaliza código nem usa hashing semântico — apenas evita loops baseados em contagem de repetição simples, se tanto.
- **FINDING:** Para hashing de AST, recomenda-se usar bibliotecas como `tree-sitter` ou `babel` para gerar uma estrutura canônica do código (por exemplo, mesmo código com nomes de variáveis trocados produzem ASTs idênticos salvo `Identifier`). Em última análise, o padrão seria “gerar hash do AST” (ignorando ordenação de import e comentários). Não há implementação pronta encontrada para TypeScript, mas acadêmicos sugerem usar hash de estrutura (por exemplo, calcular um fingerprint a partir de pré-ordem do AST). Nenhum código prático foi achado para uso imediato; a solução exigirá implementação personalizada de serialização canônica do AST.

Vulnerabilidade 7 – Vazamento de Listeners SSE

- **FINDING:** Um padrão recomendado no Node.js é usar **AbortController** no handler HTTP. Por exemplo, ao iniciar uma rota SSE pode-se fazer `const ac = new AbortController(); req.on('close', () => ac.abort());` e passar `ac.signal` para qualquer operação assíncrona (como fetch). Assim, quando o cliente desconecta, o request dispara `close` abortando pendências【127†L241-L249】. Este padrão evita acumulacão de tarefas e listeners sobrando, pois todos dependem do mesmo AbortSignal. (Veja \[127\] linhas241-249.)
- **FINDING:** Plugins SSE como `@fastify/sse` oferecem ganchos de cleanup. No exemplo oficial, após iniciar um intervalo periódico para “ping”, é registrado `reply.sse.onClose(() => { clearInterval(interval); console.log('Connection closed') })`【129†L457-L461】. Ou seja, o método `reply.sse.onClose` registra um callback que é executado quando o cliente desconecta, permitindo remover listeners ou timers. No código citado, assim que a conexão fecha, o intervalo é limpo e os listeners internos são descartados, prevenindo o acúmulo de handlers duplicados.
- **FINDING:** Não foi encontrado código padrão que automaticamente remova todos os listeners de um EventEmitter para cada reconexão. Em geral, a limpeza deve ser feita manualmente (via `removeListener` ou `removeAllListeners`) no evento de fechamento. Procurou-se por implementações que usem `once` para listeners de uma só execução ou `removeListener` em `req.on('close')`, mas nada específico foi achado em bibliotecas estudadas. Portanto, a prática é usar `req.on('close')` (ou `reply.sse.onClose`) para chamar explicitamente `off()` ou limpar recursos; não existe solução pronta que faça isso silenciosamente.

Vulnerabilidade 8 – Injeção de Expressão YAML / RCE

- **FINDING:** **CVE-2024-4264 (litellm)** – Um exemplo real de RCE por expressão insegura foi relatado no LitellM. O método `litellm.get\_secret()` usava `eval()` em dados não sanitizados de um endpoint `/config/update`, permitindo que variáveis de ambiente maliciosas alterassem o YAML de configuração (`proxy\_server\_config.yaml`) e executassem código arbitrário【135†L126-L133】. Este caso demonstra que usar `eval()` em campos de configuração YAML habilita RCE.
- **FINDING:** **CVE-2025-68613 (n8n)** – O n8n (plataforma de workflows) tinha uma RCE crítica via injeção de expressões em seu motor de avaliação de regras. Usuários autenticados podiam inserir expressões JavaScript maliciosas que escapavam da sandbox e executavam comandos no servidor【134†L7-L15】. O comunicado afirma explicitamente “Remote Code Execution via Expression Injection” no mecanismo de avaliação do servidor【134†L7-L15】, confirmando que avaliações de expressões devem ser estritamente isoladas.
- **FINDING:** **CVE-2026-39842 (OpenRemote)** – Um advisory do GitHub detalha duas vulnerabilidades em um mecanismo de regras (Nashorn/Groovy). Em Java, o código chamava diretamente `scriptEngine.eval(script)` sem `ClassFilter` nem sandboxing【148†L219-L227】. O trecho destaca `scriptEngine.eval(script, engineScope)` e ausência de restrição, permitindo `Java.type()` e chamadas perigosas (RCE). Este é outro caso onde “inserir scripts no YAML/JSON de regras” leva a execução de código.
- **FINDING:** Não foi encontrado um padrão declarado em frameworks AI para avaliação segura de expressões em YAML. As opções conhecidas incluem bibliotecas de sandbox JavaScript (ex: [shadowrealm](https://notebooklm.google.com/#) ou ), mas o estado atual dessas propostas no Node.js é experimental. A abordagem recomendada seria migrar `activation\_trigger` para um formato declarativo (por exemplo, uma lista de condições permitidas) ou usar um avaliador de expressões restrito (ex: biblioteca de parse sem `eval()`). Até agora, não há implementação padrão em projetos de agentes estudados; todos os casos de vulnerabilidade indicam uso de `eval`/`Function` inseguro. Portanto, a solução segura exigiria substituir `eval()` por um parser de expressão sandboxed ou por lógica condicional declarativa.

SÍNTESE

- **Vuln 1:** Uso de **replay de SSE baseado em IDs** (como no plugin Fastify SSE) se destacou. Esse padrão mantém histórico de eventos e, ao reconectar, reenvia apenas os eventos após o último ID recebido【129†L468-L478】. A vantagem é resolver automaticamente gaps e reconexões de SSE, alinhando streams independentes por ID. (Recomendação: implementar mecanismo similar de replay + espera de missing antes de avançar.)
- **Vuln 3:** Nenhum padrão existente foi encontrado; isso indica que a melhor solução é **introduzir locks ou contadores** entre GarbageCollector e RollbackManager. Em outras palavras, implementar contagem de referências ou marcação ativa para impedir GC prematuro. (Ou seja, deve-se criar esse mecanismo internamente.)
- **Vuln 5:** A melhor abordagem é **normalizar o código via AST** antes de calcular hashes. Em vez de hash do texto, converter para AST canônico (removendo comentários, fixando ordem de imports, renomeando identificadores) e então fazer hash da árvore estrutural. Não há biblioteca pronta em TypeScript, mas esse padrão semântico (hash de AST) é recomendado para detecção de loops.
- **Vuln 7:** O padrão mais robusto é combinar **AbortController + onClose**. Ao criar uma conexão SSE, instancie um `AbortController` e invoque `ac.abort()` em `req.on('close')`; use `reply.sse.onClose()` para limpar timers/listeners【127†L241-L249】【129†L457-L461】. Isso garante que todos os handlers ligados à conexão sejam cancelados ao desconectar, evitando leaks.
- **Vuln 8:** Como evidenciado pelos CVEs acima, **não usar****eval()** é crítico. Em vez disso, usar avaliadores de expressão sandboxed ou declarativos. A lição central (de CVE-2024-4264 e CVE-2025-68613) é que qualquer campo de configuração avaliado dinamicamente deve ser isolado ou substituído. O padrão “mais seguro” seria usar uma avaliação restrita (por exemplo, ou com limites de CPU/memória) ou converter `activation\_trigger` em uma condição declarativa simples, eliminando `eval()`.
